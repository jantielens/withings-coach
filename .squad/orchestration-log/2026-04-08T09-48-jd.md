### Data Volume Analysis

| Scenario | Readings/Year | Tokens | Context Usage |
|----------|--------------|--------|---------------|
| Light (1/day) | 365 | ~8,100 | 6% |
| Standard (2/day) | 730 | ~15,400 | 12% |
| Heavy (2/day + diary) | 730 | ~20,900 | 16% |
| 2 years of data | 1,460 | ~30,000–41,000 | 23–32% |

**Context window capacity:** 128K tokens  
**Finding:** Even 2 years of intensive monitoring fits comfortably with headroom for conversation history.

### Conversational Prompt Design

- **System prompt architecture:** Role definition + ESC/ESH reference + data tables + disclaimers + Dutch language support
- **Add for chat context:** Conversational instructions, date awareness, scope boundaries
- **Remove:** Rigid 6-section analysis template
- **Dynamic:** Rebuild system message on EVERY request to include latest data

### Data Flow

1. User sends message via chat panel
2. Frontend (useChat hook) POSTs to `/api/chat` with full conversation history
3. API route detects time intent from latest message ("last week" → 7 days)
4. Fetch BP readings, diary entries, context notes from DB
5. Build dynamic user context string (markdown table format)
6. Construct messages array: [system prompt, user context, ...conversation]
7. Call Azure OpenAI with streaming
8. Vercel AI SDK converts to SSE
9. Frontend renders tokens in real-time

### Token Management

- Cap message history sent to API at last 20 messages
- Implement `MAX_READINGS` constant (default 1000)
- If exceeded, use most recent N readings + note "Showing most recent {N} of {total}"
- Monitor request volume (estimate $1–3/month for single user)

### Model Selection

**Recommended: GPT-4o-mini**
- 128K context window
- Outperforms GPT-3.5 on all benchmarks
- ~60% cheaper than GPT-3.5
- Lower latency
- Ideal for MVP chatbot
