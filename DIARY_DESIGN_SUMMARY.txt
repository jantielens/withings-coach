TASK: Design storage architecture for per-day diary entries (free-form notes to 
enrich BP trend analysis). Address 6 key design areas.

───────────────────────────────────────────────────────────────────────────────
1. STORAGE RECOMMENDATION: SQLite
───────────────────────────────────────────────────────────────────────────────

✅ SELECTED: SQLite (file-based, zero config, ships with Node.js)

WHY:
  • MVP Simplicity: Single .db file, no infrastructure setup
  • Migration Path: userId column ready from day one (scales to multi-user without refactor)
  • Zero Cloud Costs: $0 for MVP
  • Next.js Native: Works seamlessly with Node.js backend routes
  • Backup/Export: One portable file
  • Concurrent Access: Safe with WAL mode

REJECTED ALTERNATIVES:
  • JSON file → Multi-user JSON requires per-user files (nightmare) or nested 
    structure (unqueryable). SQLite wins on scalability.
  • localStorage → Browser-only. No server sync, no backup, dead end for any 
    future server features.
  • Better-sqlite3/Prisma → Unnecessary complexity for MVP. Prisma adds 50+ 
    lines of boilerplate for a 3-table schema.
  • Azure Table Storage → Wins on multi-user readiness but adds cloud billing 
    ($$/month) and vendor lock-in. Save for Phase 2 if needed.

IMPLEMENTATION: npm install sqlite3
  Use native Node driver. Optional: wrap with better-sqlite3 later if perf issues 
  emerge (they won't in MVP).

───────────────────────────────────────────────────────────────────────────────
2. DATA MODEL: Minimal Entry Schema
───────────────────────────────────────────────────────────────────────────────

✅ ENTRY STRUCTURE:
  {
    id: string                    // UUID: "diary_{userId}_{date}"
    userId: string                // Future: multi-user support (hardcoded "default" in MVP)
    date: string                  // ISO 8601 date (YYYY-MM-DD) — diary is per-day
    text: string                  // Free text (max 1000 chars)
    createdAt: string             // ISO 8601 timestamp
    updatedAt: string             // ISO 8601 timestamp
  }

✅ UNIQUENESS CONSTRAINT: UNIQUE(userId, date)
  One entry per day per user. Simplifies edit logic: either update existing or 
  create new, never merge.

✅ MINIMAL DESIGN (KISS):
  NO category/tags/mood/severity in MVP. The text IS the context. LLM can parse 
  mood ("stressful") or medication ("took ABC") from free text in Phase 2.
  
  NO time-of-day granularity. Diary is per-day. "took medicine ABC" doesn't need 
  a timestamp.

✅ DATABASE SCHEMA:
  CREATE TABLE IF NOT EXISTS diary_entries (
    id TEXT PRIMARY KEY,
    userId TEXT NOT NULL,
    date TEXT NOT NULL,
    text TEXT NOT NULL,
    createdAt TEXT NOT NULL,
    updatedAt TEXT NOT NULL,
    UNIQUE(userId, date)
  );
  CREATE INDEX idx_diary_userId_date ON diary_entries(userId, date);

───────────────────────────────────────────────────────────────────────────────
3. API DESIGN: Simple REST for Diary
───────────────────────────────────────────────────────────────────────────────

✅ ROUTES:

  GET /api/diary?date=2025-07-15&userId=default
    → Returns one entry for date, or 404 if none

  GET /api/diary?from=2025-07-01&to=2025-07-31&userId=default
    → Returns all entries for date range (for LLM prompt)

  POST /api/diary
    { userId: "default", date: "2025-07-15", text: "Took medicine ABC..." }
    → Creates or updates entry for that date (UNIQUE constraint prevents dupes)

  DELETE /api/diary?date=2025-07-15&userId=default
    → Deletes entry for date

✅ SEPARATION OF CONCERNS:
  Diary is SEPARATE from health metrics in API and storage:
    • Health metrics: Withings API → HealthDataService → /api/health/metrics
    • Diary: SQLite → DiaryService → /api/diary
  
  INTEGRATION happens at presentation layer (prompt builder + UI components), not 
  API layer. This keeps code modular and testable.

6. MIGRATION PATH: Single-User MVP → Multi-User
───────────────────────────────────────────────────────────────────────────────

✅ MVP (NOW):
  • Single user: userId = "default" (hardcoded in API routes, env var in UI)
  • No auth: Same static token as health metrics
  • Data structure ready: userId column exists but unused

  Code pattern:
  const userId = process.env.DIARY_USER_ID || 'default';
  const entry = await diaryService.getEntry(userId, date);

✅ PHASE 2 (MULTI-USER):
  1. Add Auth: Use same AuthProvider interface, implement OAuth
  2. Extract userId: From JWT/session instead of hardcoding
  3. Add WHERE clause: Already in schema and API (no refactoring)
  4. Optional cloud migration: If scaling, dump to Azure Table Storage

  Code pattern (same API signature):
  const userId = extractUserIdFromAuth(auth);  // OAuth or session
  const entry = await diaryService.getEntry(userId, date);  // Identical call

✅ WHY ZERO BREAKING CHANGES:
  • userId in all queries from day one → multi-user just adds auth extraction
  • Storage-agnostic schema → SQLite or Azure, same API
  • Auth-agnostic → Static token or OAuth, same userId extraction point
